# Архитектура учебной библиотеки генерации тестовых данных
## Назначение проекта
Проект является учебным проектным интенсивом по программированию.
В рамках проекта ученики создают библиотеку для генерации тестовых данных в двух вариантах:
1. Процедурная версия.
2. Объектно-ориентированная версия.
Обе версии библиотеки должны работать одинаково.
Разница между ними заключается только в стиле программирования и организации кода.
Процедурная версия помогает ученикам закрепить работу с функциями, модулями и разделением кода по файлам.
ООП-версия помогает ученикам понять классы, объекты, методы, состояние объекта и распределение ответственности между классами.
Проект также знакомит учеников с настоящей задачей генерации тестовых данных.
Генераторы должны уметь создавать не только случайные значения, но и данные под конкретные проверки.
Например, строку ровно на `255` символов, возраст на нижней границе, email с ошибкой формата или список из нужного количества уникальных тегов.
## Главный архитектурный принцип
Проект состоит из двух независимых реализаций одной и той же библиотеки.
Обе реализации должны предоставлять одинаковые возможности и возвращать данные в одинаковом формате.
Если в процедурной версии есть функция для генерации имени, то в ООП-версии должен быть метод, который решает ту же задачу.
Если в процедурной версии есть функция для генерации полного имени, то в ООП-версии должен быть метод, который возвращает данные такого же вида.
Если процедурная функция поддерживает параметры для границ, длины или валидности, то ООП-метод должен поддерживать такие же параметры.
Так ученики видят, что одинаковое поведение можно организовать разными стилями программирования.

## Процедурная версия
Папка `procedural_version/` содержит реализацию библиотеки в процедурном стиле.
Основная единица кода в этой версии — функция.
Код разделяется по файлам в зависимости от типа генерируемых данных.
### Модули процедурной версии
В процедурной версии каждый генератор находится в отдельном файле.
Файлы генераторов:
- `generators/user_id.py` — `user_id()`;
- `generators/age.py` — `age()`;
- `generators/birth_year.py` — `birth_year()`;
- `generators/comment.py` — `comment()`;
- `generators/password.py` — `password()`;
- `generators/phone.py` — `phone()`;
- `generators/email.py` — `email()`;
- `generators/city.py` — `city()`;
- `generators/score_example.py` — короткое имя `score_example()`, функция-образец;
- `generators/active_example.py` — короткое имя `active_example()`, функция-образец;
- `generators/tags.py` — `tags()`;
- `generators/plan_example.py` — короткое имя `plan_example()`, функция-образец;
- `generators/reg_date_example.py` — короткое имя `reg_date_example()`, функция-образец;
- `generators/user_profile.py` — `user_profile()`;
- `generators/first_name.py` — `first_name()`;
- `generators/last_name.py` — `last_name()`;
- `generators/full_name.py` — `full_name()`;
- `generators/username.py` — `username()`.
`data/` содержит исходные списки данных.
`utils/` содержит общие вспомогательные функции.
## ООП-версия
Папка `oop_version/` содержит реализацию той же библиотеки в объектно-ориентированном стиле.
Основная единица кода в этой версии — класс.
Классы должны повторять возможности процедурных модулей.
### Классы ООП-версии
`BaseGenerator` содержит общую логику, которая нужна разным генераторам.
`PersonGenerator` отвечает за генерацию основных данных человека.
Методы:
- `user_id()`;
- `age()`;
- `birth_year()`;
- `city()`;
- `score_example()`;
- `active_example()`;
- `first_name()`;
- `last_name()`;
- `full_name()`;
- `username()`.
`ContactGenerator` отвечает за генерацию контактных данных.
Методы:
- `phone()`;
- `email()`.
`TextGenerator` отвечает за генерацию текстовых данных.
Методы:
- `comment()`;
- `password()`;
- `tags()`.
`ProfileGenerator` отвечает за генерацию профиля пользователя.
Методы:
- `reg_date_example()`;
- `plan_example()`;
- `user_profile()`.
`DataProvider` отвечает за хранение и выдачу исходных списков данных.

## Правило одинакового поведения
Обе версии должны генерировать данные по одинаковым правилам.
Например:
- ID пользователя должен быть целым числом в одинаковом диапазоне;
- возраст должен быть целым числом в одинаковом диапазоне;
- год рождения должен быть целым числом в одинаковом диапазоне;
- комментарий должен быть длинной строкой;
- пароль должен быть строкой в одинаковом формате;
- телефон должен быть словарем с ключами `country_code`, `operator_code`, `number`;
- email должен быть строкой в одинаковом формате;
- город должен выбираться из одного и того же списка;
- балл должен быть целым числом в одинаковом диапазоне;
- активность должна быть булевым значением;
- теги должны быть списком строк;
- план подписки должен выбираться из одного и того же списка;
- дата регистрации должна быть строкой в одинаковом формате;
- профиль пользователя должен быть словарем;
- имя должно выбираться из одного и того же списка;
- фамилия должна выбираться из одного и того же списка;
- полное имя должно собираться по одному и тому же принципу;
- username должен быть строкой в одинаковом формате;
`seed` помогает получать одинаковый случайный результат. Если вызвать генератор два раза с одинаковым `seed`, результат должен повториться. Если `seed=None`, результат может быть разным при каждом запуске.
## Учебная логика проекта
Проект должен идти по этапам.
1. Ученики создают процедурную версию библиотеки.
2. Ученики запускают готовые тесты для процедурной версии.
3. Ученики создают ООП-версию библиотеки.
4. Ученики запускают готовые тесты для ООП-версии.
5. Ученики сравнивают поведение двух версий.
### Этап 1. Процедурная версия
Сначала ученики создают процедурную версию.
На этом этапе они изучают:
- функции;
- параметры функций;
- возвращаемые значения;
- импорт из других файлов;
- разделение проекта на модули;
- работу со списками;
- работу со случайными значениями;
- работу с граничными значениями;
- генерацию строк заданной длины;
- генерацию валидных и невалидных данных.
### Этап 2. Проверка процедурной версии тестами
После реализации процедурной версии ученики запускают готовые тесты для ее функций.
На этом этапе они изучают:
- зачем нужны автоматические тесты;
- как запускать тесты из командной строки;
- как читать сообщение об ошибке;
- как понимать, какая функция работает неправильно;
- как находить ошибки в процедурном коде;
- как проверять требования к длине, диапазону и формату данных.
### Этап 3. ООП-версия
После тестирования процедурной версии ученики создают ООП-версию.
На этом этапе они изучают:
- классы;
- объекты;
- методы;
- атрибуты;
- наследование;
- композицию;
- повторное использование кода;
- распределение ответственности между классами.
### Этап 4. Проверка ООП-версии тестами
После реализации ООП-версии ученики запускают готовые тесты для ее методов.
На этом этапе они изучают:
- как запускать тесты для ООП-кода;
- как читать ошибки, связанные с классами и методами;
- как понимать, какой метод работает неправильно;
- как находить ошибки в объектно-ориентированном коде;
- как сохранять одинаковое поведение при другой организации кода.
### Этап 5. Сравнение версий
После того как процедурная и ООП-версия по отдельности проходят свои проверки, ученики сравнивают их поведение.
На этом этапе важно проверить, что при одинаковых параметрах и одинаковом `seed` обе версии возвращают одинаковые результаты.
На этом этапе важно убедиться, что различается только стиль программирования, а не результат работы библиотеки.
## Польза проекта для знакомства с автотестами
Проект помогает ученикам познакомиться с применением автотестов на понятной учебной задаче.
Ученики видят, что тесты проверяют не сам факт наличия кода, а поведение функций и методов.
Автотесты показывают, какой результат ожидается от генератора данных.
Если реализация написана неправильно, тест помогает быстро найти проблемную функцию или метод.
Так ученики учатся воспринимать тесты не как наказание за ошибку, а как инструмент обратной связи.
В проекте автотесты помогают проверить процедурную версию и ООП-версию.
Это показывает важную идею: код может быть организован по-разному, но внешнее поведение должно оставаться одинаковым.
Благодаря этому ученики постепенно привыкают к рабочему процессу, который используется в реальной разработке:
- написать или изменить код;
- запустить тесты;
- прочитать результат проверки;
- исправить ошибку;
- снова запустить тесты.
## Тестирование
Для проекта важно иметь готовые тесты для обеих версий.
Тесты предоставляет преподаватель.
Ученики не обязаны писать тесты самостоятельно.
Ученики используют тесты как инструмент самопроверки после выполнения каждого этапа.
Тесты должны проверять не внутреннее устройство кода, а внешнее поведение библиотеки.
Проверки должны отвечать на вопрос: одинаково ли работают две версии с точки зрения пользователя библиотеки.
Что нужно проверять:
- возвращаемый тип данных;
- попадание ID пользователя в заданный диапазон;
- попадание возраста в заданный диапазон;
- попадание года рождения в заданный диапазон;
- достаточную длину комментария;
- формат пароля;
- структуру словаря с номером телефона;
- формат email;
- попадание города в список городов;
- попадание балла в заданный диапазон;
- булевый тип активности;
- структуру списка тегов;
- попадание плана подписки в список планов подписки;
- формат даты регистрации;
- структуру словаря профиля пользователя;
- формат полного имени;
- формат username;
- наличие одинакового публичного набора возможностей в обеих версиях.
## Итоговая цель
К концу интенсива ученики должны получить две версии одной библиотеки:
- библиотеку в процедурном стиле;
- библиотеку в ООП-стиле.
Обе версии должны решать одну и ту же задачу и быть удобными для сравнения.
Главная ценность проекта не в количестве генераторов, а в понимании того, как одна и та же логика может быть организована разными стилями программирования.
